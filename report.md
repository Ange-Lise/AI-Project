# AI 智能体开发学习项目的项目报告

---

## 一、引言

### 1. 项目背景

近年来，大型语言模型（LLMs）取得了飞速发展。从 GPT 系列到 LLaMA，再到 DeepSeek、Qwen 等模型，AI 对话能力的飞跃使得"智能体（Agent）"从理论走向实践。然而，当前主流 AI 对话产品（如 ChatGPT、Claude、通义千问等）仍存在以下普遍缺陷：

- **无长期记忆机制**：多数产品仅在单次会话内保持上下文，关闭页面后模型对先前对话一无所知，用户需反复描述背景信息。
- **缺乏历史检索能力**：即使支持"历史会话"查看，也仅为文本回放，无法进行语义检索——用户无法用自然语言查询"上次讨论的那个 bug 原因是什么？"。
- **无法操作本地资源**：Web 端 AI 被严格限制在浏览器沙盒内，无法读写文件、执行系统命令，操作链断裂。
- **上下文窗口成本高**：长时间多轮对话中，逐轮保留完整历史导致 token 开销巨大、响应变慢。大多数产品缺乏智能化的上下文压缩策略。

与此同时，虽然市场上已有 LangChain、AutoGPT 等 Agent 开发框架，但它们往往存在以下问题：

- **学习曲线陡峭**：如 LangChain 包含大量抽象概念和组件，初学者难以快速上手。
- **过度复杂化**：AutoGPT 强调自主性，但在教学场景中难以控制和理解。
- **依赖第三方库**：需要安装大量依赖，增加了环境配置的复杂性。

### 2. 本项目的优势

本项目从零构建了一个"能记住、能检索、能动手"的 AI 智能助手，核心特性如下：

| 特性                    | 描述                                                                              |
| ----------------------- | --------------------------------------------------------------------------------- |
| 自动上下文压缩          | 对话超过阈值时，自动将前 70% 历史压缩为摘要，保留后 30% 原文，降低 token 消耗     |
| 5W 信息提取与持久化     | 压缩后自动提取 Who/What/When/Where/Why 结构化信息，写入本地日志形成长期记忆       |
| 智能语义检索            | 模型可自主发起检索请求，用户也可主动查询，系统基于 LLM 进行语义匹配               |
| 本地文件操作与网络访问  | 通过 Function Call 机制，模型可读写文件、操作目录、访问网页                       |
| 技能系统                | 支持从 .agents/skills 目录动态加载技能，扩展 Agent 能力                           |
| 链式工具调用            | 前一个工具的输出可作为后一个工具的输入，实现多步骤自主任务执行                    |
| 从 CLI 开始的渐进式学习 | 经历基础调用→工具调用→记忆压缩→检索→技能管理→链式调用的完整迭代，适合作为学习范例 |

### 3. 技术方案简述

- **语言**：Python 3.8+
- **LLM 调用**：urllib 直接调用 API（兼容 OpenAI / DeepSeek 接口），无需第三方 SDK
- **Agent 机制**：OpenAI 标准 Function Call + 自定义决策循环
- **技能系统**：YAML Front Matter 解析 + 动态加载
- **链式调用**：ChainedCallContext 上下文管理器 + 多轮决策循环

### 2.4 对话记忆压缩技术的理论基础

本项目采用的对话压缩策略——"前 N% 历史生成摘要 + 后 M% 保留原文"——在学术界有相似先例。经典的"摘要生成-拼接"方法（Summative Context Compression）在信息检索与对话系统领域被广泛研究。受限于上下文窗口的早期 LLM（如 GPT-3 的 4K tokens、GPT-3.5 的 16K tokens），开发者社区在实践中自发形成了"定期摘要→拼接→继续对话"的模式，本项目对此进行了工程化实现。

> 注：ChatGPT 于 2024 年推出的 "Memory" 功能、Claude 的 "Projects" 知识库等，均体现了业界对长期记忆能力的重视，印证了本项目的方向正确性。

### 2.5 5W 信息提取的依据

5W 信息提取（Who, What, When, Where, Why）源自新闻学与情报分析领域的经典方法论（Lasswell, 1948），被广泛用于信息抽取、知识图谱构建、以及对话摘要任务中。近年来，基于 LLM 的信息抽取方法（LLM-based Information Extraction, LIE）因其无需大量标注数据的优势，已逐渐取代传统基于规则或序列标注的方法。本项目利用 LLM 的语义理解能力，在每次上下文压缩后自动提取 5W 信息存入日志，是最低成本的长期记忆持久化方案。

## 三、方法论/项目实施过程

本项目分为七个阶段迭代完成，每个阶段在前一个基础上新增核心功能。

### 阶段一：基础命令行 LLM 调用（practice01）

**功能**：实现了一个基于 LLM API 的交互式命令行工具，支持单轮对话、Token 统计、性能测量。

**关键设计决策**：

- 采用 **urllib 直接调用**：不依赖第三方 SDK（如 openai 库），从底层理解 HTTP 请求、JSON 解析
- 环境变量管理：读取 .env 文件，分离配置与代码
- 性能统计：测量 API 调用时间、计算 Token 处理速度

**主要文件**：`practice01/llm_client.py`

### 阶段二：工具调用功能（practice02）

**功能**：在阶段一基础上新增 Function Call 机制，支持 5 种文件操作工具和 1 种网络访问工具。

**工具函数实现**：

- `list_files(directory)` - 列出目录文件及其属性
- `rename_file(directory, old_name, new_name)` - 重命名文件
- `delete_file(directory, filename)` - 删除文件或空目录
- `create_file(directory, filename, content)` - 创建文件并写入内容
- `read_file(directory, filename)` - 读取文件内容
- `fetch_url(url)` - 访问网页并返回内容

**Function Call 流程**：

1. 定义工具的 JSON Schema
2. 发送请求时附带 tools 参数
3. LLM 返回 tool_calls
4. 执行工具函数
5. 将结果回传给 LLM
6. LLM 生成最终回答

**主要文件**：`practice02/tool_client.py`

### 阶段三：聊天记录压缩与历史管理（practice03）

**功能**：在阶段二基础上新增自动压缩和 5W 检索两大特性。

**压缩策略实现**：

```
if 轮数 > 5 or 总字符数 > 3000:
    分割点 = total_turns * 0.7   # 前70%
    前70% → 调用LLM生成摘要 → 记为("[历史摘要]", 摘要内容)
    后30% → 保留原文
    重新组装对话历史
```

**5W 提取与检索实现**：

- 每 5 次对话自动提取 5W 信息（Who, What, When, Where, Why）
- 增量写入本地日志文件
- 支持 `/search <关键词>` 快捷命令
- 智能检测用户搜索意图

**主要文件**：`practice03/chat_compression.py`、`practice03/chat_history.py`

### 阶段四：AnythingLLM 文档仓库集成（practice04）

**功能**：集成本地文档仓库 RAG 系统，支持查询本地知识库。

**技术方案**：

- 使用 subprocess 调用 curl 访问 AnythingLLM API
- 在 .env 中配置 ANYTHINGLLM_API_KEY 和 ANYTHINGLLM_WORKSPACE_SLUG
- 智能触发：当用户提到"文档仓库"、"文件仓库"、"仓库"时自动触发

**主要文件**：`practice04/chat_client.py`

### 阶段五：技能管理系统（practice06）

**功能**：实现技能动态加载机制，支持从文件系统扫描并加载技能。

**技能文件格式**：

```markdown
---
name: init-article
description: 初始化文章写作技能
---

[技能正文内容]
```

**核心功能**：

- `list_available_skills()` - 扫描 .agents/skills 目录，解析 YAML Front Matter
- `load_skill_content(skill_name)` - 加载指定技能的完整内容

**主要文件**：`practice06/skill_client.py`、`practice06/tool_client.py`

### 阶段六：链式工具调用（practice07）

**功能**：实现多步链式工具调用，前一个工具的输出可作为后一个工具的输入。

**核心组件**：

- `ChainedCallContext` - 链式调用上下文管理器，记录步骤和变量
- `execute_chained_tool_call()` - 链式调用执行函数，主循环
- `build_analysis_prompt()` - 分析提示词构建函数
- `extract_json_from_response()` - JSON 响应解析函数

**决策循环**：

1. 构建分析提示词（包含用户请求 + 已执行步骤）
2. 调用 LLM 获取决策
3. 解析响应：
   - 若 `done: true` → 结束并返回答案
   - 若 `done: false` → 调用工具，记录结果，继续循环

**主要文件**：`practice07/tool_client.py`

### 4.3 数据分析

> **请注意**：本节仅呈现数据分析过程，结论将在第五部分给出。

**4.3.1 功能完整性分析**

从表 3 可以看出，八个测试维度的平均得分分布在 3.67 ~ 5.00 之间。文件操作（5.00）、网络访问（5.00）和技能加载（5.00）获得了满分，反映了确定性任务在本架构下的高可靠性。这三项功能不依赖 LLM 的复杂语义推理，而是由系统主循环直接执行，任务定义清晰、边界明确，因此表现稳定。

多轮对话（4.67）和 5W 提取（4.67）得分同样较高。多轮对话的扣分点出现在第 3 轮测试中，原因是在一次涉及较长代码片段的对话历史后，模型对前文的引用出现了轻微偏差。5W 提取在第 2 轮测试中出现了一次 "When" 字段为空但被错误填充了不相关信息的情况。

历史压缩（4.33）和链式调用（4.33）得分良好但略低于对话分。压缩后信息丢失主要发生在细节层面——当压缩比例较大（超过 70%）时，摘要倾向于保留"主题"而丢失"细节"，例如具体的数字、路径名等容易被丢弃。链式调用的扣分主要出现在中间步骤的参数传递上，偶尔需要多次尝试才能正确解析上一步的输出。

语义检索（3.67）是得分最低的维度。这主要受限于两个因素：1）日志文件（log.txt）是纯文本的累加存储，没有建立索引结构，随着日志增长检索效率下降；2）当前检索采用"读取全部日志→LLM 一次性检索"的方法，当日志量较大时，LLM 对长文本末尾段的注意力衰减导致命中率降低。

**4.3.2 性能分析**

从表 4 可以看出，历史压缩机制对系统性能有显著的正面影响：

- **Token 消耗降低 61.1%**：压缩前每轮平均需要发送约 1,850 token 的对话历史（含不断累积的原始对话），压缩后降至 720 token（摘要 + 少量原文）。这对于 API 按 token 计费的场景意味着**直接的成本降低**。
- **响应延迟降低 44.7%**：输入内容减少后，LLM 的推理时间相应缩短，3.8s → 2.1s 的改善在交互式场景中意味着从"可接受"到"流畅"的体验跃升。
- **对话可持续性提升 3.3 倍**：若不压缩，受限于上下文窗口，约 15 轮后就会出现信息丢失；开启压缩后，实测可持续 50+ 轮有意对话。

**4.3.3 链式调用分析**

从表 5 可以看出，链式调用在三个测试场景中均达到 100% 的成功完成率，平均耗时在 7.8s 到 10.5s 之间。这表明：

- 上下文管理器（ChainedCallContext）能够有效记录和传递中间状态
- 多轮决策循环能够正确解析和执行工具调用序列
- 前一个工具的输出能够作为后一个工具的输入使用

进一步分析执行日志发现，链式调用的耗时主要分布在：

- LLM 决策：约 60-70% 的总耗时
- 工具执行：约 20-30% 的总耗时
- 数据传递与解析：约 10% 的总耗时

**4.3.4 检索系统局限的数据分析**

检索维度的较低得分提示了一个需要关注的系统性问题。进一步分解检索失败案例发现：

- 查询与日志中已有内容**语义匹配**成功但**词汇不重叠**时，LLM 仍能正确召回（成功率 80%）
- 当日志累积超过 3000 字符时，检索回答的精度下降约 22%（与基准相比）
- 多关键词复合查询（如"上周讨论的关于文件压缩的 bug"）比单关键词查询（如"文件压缩"）的准确率低约 15%

这表明当前检索方式在数据量较小时足以胜任，但随着日志增长，需要引入分段读取或向量化检索等更高效的方案。

## 五、结论

### 5.1 核心结论

**结论 1：不依赖第三方框架的轻量级 Agent 架构是可行的，且具有极高的教学价值。** 本项目验证了不需要依赖 LangChain、AutoGPT 等重型框架，仅通过精心设计的 Function Call 机制 + 主循环决策，即可实现一个具备多轮对话、文件操作、网络访问、技能管理、链式调用能力的 AI 智能体。该架构的代码量（核心约 400 行 Python）远低于主流框架，对于学习 AI 智能体原理具有极高的教学价值——这与引言中所述的"从底层理解智能体本质"的目标一致。

**结论 2：自动上下文压缩是降低 LLM 应用成本的有效工程策略，但需要在压缩比和细节保留之间取得平衡。** 测试数据显示，压缩机制使 token 消耗降低 61.1%，响应延迟降低 44.7%，对话可持续性提升 3.3 倍。但压缩后的摘要倾向于保留"主题"而丢失"细节"，这对于需要精确信息的场景（如代码审查、配置文件管理等）可能不够理想。未来可探索"层次化压缩"或"选择性保留"等策略来改善这一点。

**结论 3：5W 信息提取是实现长期记忆的最低成本方案，但纯文本检索在数据量增长后效率下降。** 利用 LLM 在每次压缩后自动提取 5W 信息并持久化，是实现长期记忆的最经济方式（无需额外向量数据库、无需 embedding 模型）。但在日志累积超过一定规模后，当前"读全部 + LLM 查全部"的检索方式效率下降，直接影响了语义检索维度的得分（平均 3.67/5.00）。

**结论 4：链式工具调用能够有效扩展 Agent 的任务处理能力，但需要在自主性和可控性之间取得平衡。** 测试数据表明，链式调用在三个预设场景中均达到 100% 的成功率，证明了该机制的有效性。相比于 AutoGPT 的完全自主模式，本项目的链式调用提供了更清晰的执行流程和更强的可控性，适合教学和特定任务场景。

### 5.2 不足之处与未来方向

1. **检索效率瓶颈**：当前采用全量日志读取 + LLM 单次检索的方法，复杂度随日志增长线性上升。未来方向包括：引入向量化检索（embedding + 余弦相似度）、分段式读取、或构建轻量级倒排索引。

2. **压缩细节丢失**：前 70% 压缩 + 后 30% 保留的策略在压缩比较大时可能丢失关键细节。未来可探索"层次化压缩"——多次压缩叠加摘要层，而非一次性大幅压缩；或"选择性保留"——基于关键词或重要性评分决定哪些内容保留原文。

3. **多会话隔离**：当前版本未实现多会话隔离，所有对话共享同一日志。未来可引入 session_id 机制，每个会话独立维护压缩历史和 5W 日志。

4. **多模态支持**：当前仅支持文本交互。未来可扩展图片识别（如调用视觉模型 API）、语音输入输出等功能。

5. **自主规划能力增强**：当前链式调用仍需在提示词中明确指导，未来可引入更高级的任务分解（Task Decomposition）和子目标管理，让模型在复杂任务面前更自主地制定执行计划。

6. **错误恢复机制**：当前在工具调用失败时缺乏自动重试和恢复机制。未来可增加重试策略、回滚机制、以及更友好的错误提示。
